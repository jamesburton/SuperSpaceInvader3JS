**ShopUI changes:**

Add cursor selection state for gamepad/keyboard arrow navigation:
- New private field: `selectedIndex: number = 0`
- New public method `update(input: InputManager): void`:
  - If not visible, return early
  - Get items from `this.getItemsFn?.() ?? []`; if empty, return
  - If `input.justPressed('ArrowUp')`: decrement selectedIndex with wrap `(selectedIndex - 1 + items.length) % items.length`, call `this._render()`, play `audioManager.playSfx('menuNav')`
  - If `input.justPressed('ArrowDown')`: increment selectedIndex with wrap `(selectedIndex + 1) % items.length`, call `this._render()`, play `audioManager.playSfx('menuNav')`
  - If `input.justPressed('Space')`: call `this._buyItem(items[this.selectedIndex])` (A button = Space = confirm/purchase)
  - If `input.justPressed('Escape')`: call `this.onCloseFn?.()` (B button = Escape = close)
  - Note: digit-key purchases and existing Escape keyHandler still work via the DOM keydown listener — this update() method adds arrow/button navigation on top
- Import `InputManager` type at top: `import type { InputManager } from '../core/InputManager';`
- Modify `_render()` to show selection highlight:
  - Pass `this.selectedIndex` into the row rendering
  - For the selected row, add a left-side indicator: a bright cyan left-border (e.g., `border-left: 3px solid #00ffff`) and slightly brighter background (`rgba(0,255,255,0.08)`)
  - Non-selected rows have transparent left-border
  - This highlight is always visible (not just when gamepad active) — keyboard arrow navigation benefits too
- Modify `_render()` hint text to be dynamic: change the static "1-9 to buy | ESC to close" line based on `this.lastInputDevice` (store from update calls or read from a passed InputManager). Simplest approach: store a reference. Actually simplest: just always show both hints side by side since both keyboard and gamepad work simultaneously. Change to: `"1-9 / D-PAD to select | A to buy | B / ESC to close"`
- In `_buyItem()`, after purchase succeeds and re-renders, clamp selectedIndex: `this.selectedIndex = Math.min(this.selectedIndex, items.length - 1)` (Pitfall 5 from research — prevents cursor out-of-bounds if items array shrinks)
- In `show()`, reset `this.selectedIndex = 0` so cursor starts at first item

**MetaShopUI changes:**

MetaShopUI is more complex — it uses a card grid layout with multiple categories, not a simple list. The D-pad navigation needs to work across a flat list of purchasable (non-owned, non-locked) cards.

Add cursor navigation:
- New private field: `selectedIndex: number = 0`
- New private field: `purchasableIds: string[] = []` — flat ordered list of purchasable upgrade IDs (non-owned, non-locked), rebuilt each render
- New public method `update(input: InputManager): void`:
  - If not visible, return early
  - If purchasableIds is empty, check only Escape for close
  - If `input.justPressed('ArrowUp')` or `input.justPressed('ArrowLeft')`: decrement with wrap, re-render, play menuNav SFX
  - If `input.justPressed('ArrowDown')` or `input.justPressed('ArrowRight')`: increment with wrap, re-render, play menuNav SFX
  - If `input.justPressed('Space')`: trigger purchase of `purchasableIds[selectedIndex]` — call the same logic as `window.__metaShopBuy(id)` (extract purchase logic into a private `_buyById(id)` method)
  - If `input.justPressed('Escape')`: call `this.onCloseCallback?.()`
- Import `InputManager` type at top
- Modify `render()`:
  - Build `this.purchasableIds` array during rendering by collecting IDs of upgrades that are not owned and not locked (in render order)
  - For the card matching `purchasableIds[selectedIndex]`, add a highlight style: brighter border (white or bright cyan), box-shadow glow. Use a CSS class or inline style conditional.
  - Clamp selectedIndex after rebuilding purchasableIds: `this.selectedIndex = Math.min(this.selectedIndex, Math.max(0, this.purchasableIds.length - 1))`
- Modify hint text at bottom: change "ESC / U to close" to "ESC / U / B to close" (B already synthesizes as Escape, but the hint tells the player)

**GameOverState changes (src/states/GameOverState.ts):**

The enter() method builds the overlay HTML with hardcoded hint strings. Since enter() runs once and the overlay is static, and the user can switch devices while on the game over screen, the hints need to be dynamic. Two approaches:
1. Re-render on each update() if activeInputDevice changes (overkill for this screen)
2. Show both hints in compact form (simpler, always correct)
3. Update the hint DOM elements in update() when device changes

Use approach 3 — add data attributes to hint elements, update in update():

- In enter(), give hint elements IDs: `id="hint-continue"`, `id="hint-restart"`, `id="hint-menu"`
- Add private field: `private lastHintDevice: 'keyboard' | 'gamepad' = 'keyboard'`
- In update(), before the existing input checks, check if `this.input.activeInputDevice !== this.lastHintDevice`:
  - If changed, update hint text via DOM:
    - `const isGamepad = this.input.activeInputDevice === 'gamepad';`
    - Restart hint: isGamepad ? `A: ${isVictory ? 'PLAY AGAIN' : 'RESTART'}` : `PRESS R TO ${isVictory ? 'PLAY AGAIN' : 'RESTART'}`
    - Menu hint: isGamepad ? `B: MENU` : `PRESS M FOR MENU`
    - Continue hint (if visible): isGamepad ? `Y: CONTINUE` : `PRESS C TO CONTINUE`
  - Set `this.lastHintDevice = this.input.activeInputDevice`
- Store `isVictory` as a private field (already available as `this.type === 'victory'`)

Also add gamepad button handling in update():
- A button (Space) already maps to... wait, GameOverState uses KeyR for restart and KeyM for menu. A button synthesizes as Space, not KeyR. Per user decisions, GameOver: A = Restart, B = Return to Menu, Y = Continue.
  - A = Space -> needs to trigger restart. Add: `if (this.input.justPressed('Space')) { this.restartGame(); return; }` (Space is not currently used in GameOverState — safe to add)
  - B = Escape -> needs to trigger return to menu. Add: `if (this.input.justPressed('Escape')) { this.returnToMenu(); return; }` (Escape is not currently used in GameOverState — safe to add)
  - Y = KeyC -> already handled (line 81: `this.input.justPressed('KeyC')` triggers onContinue)
  - KeyR and KeyM remain functional for keyboard users (no removal)
- Order of checks in update(): Continue (KeyC) first (existing), then Space (restart via gamepad A), then KeyR (restart via keyboard), then Escape (menu via gamepad B), then KeyM (menu via keyboard)
